Kerf Build Plan — Platform Expansion

Version 2.0 — 2026-04-10

Overview

This document is the execution plan for expanding Kerf from a safety compliance platform into a graph-intelligent construction business platform. It assumes the Phase 1 safety foundation (Sprints 0-10 in BUILD_PLAN.md) is complete or near-complete, and builds on the existing knowledge graph, inspection engine, worker/equipment/project entities, conversational interface, and voice architecture.

Conversational-first: Every feature in this expansion is accessed primarily through Kerf's conversational interface (chat and voice). Forms and dashboards exist for review and bulk operations, but the contractor's primary interaction with every feature is conversational. Voice is not "where applicable" — it is the interaction model for every feature.

Strategic Context

Why This Order

Daily Logs are the Trojan horse. The contractor is already in the app for safety; daily logs make it the first thing they open every morning. 40-50% of daily log fields are already captured by safety workflows. This is the lowest-effort, highest-stickiness expansion.

Quality Inspections reuse the same engine. The inspection workflow (observe → photograph → assign corrective action → track → close) is technically identical for safety and quality. This is a template expansion, not a new product.

Time Tracking has the strongest ROI case. 38% of contractors still use paper timesheets. Time theft costs ~$52K/year per 50-person crew. And no competitor connects time data to safety data (fatigue analysis).

Sub Management extends what exists. The GC portal, prequalification automation, and compliance visibility are built. The biggest gap — COI tracking — is the highest-pain item (7/10 COIs are non-compliant).

RFIs and Scheduling are platform stickiness plays. They complete the "one tool" promise but are less urgent than the field-operations features above.

Phase 1: Daily Logs

Weeks 13-15 • ~68 hours • 2-3 weeks

Goal: The contractor's morning safety walk produces a complete daily log with 50% of fields auto-populated from existing data. Replaces the most legally important document a contractor produces.

Dependencies

  • Project entity (Sprint 2)
  • Inspection engine with photos/GPS (Sprint 2)
  • Worker tracking (Sprint 4)
  • Equipment tracking (Sprint 7/8)
  • Toolbox talks (Sprint 3)

Data Model

(:Company)-[:OWNS_PROJECT]->(:Project)-[:HAS_DAILY_LOG]->(:DailyLog {id: log_id})

Tasks

P1-T1: DailyLog backend model and service 16 hours

Create DailyLog Pydantic model, Neo4j-backed service with CRUD operations, and auto-population logic that pulls from today's inspections, toolbox talks, incidents, hazard reports, equipment assignments, and worker assignments.

Files: backend/app/models/daily_log.py, backend/app/services/daily_log_service.py

P1-T2: DailyLog API endpoints 10 hours

CRUD routes for daily logs. POST creates a draft with auto-population. PATCH updates manual fields. POST .../submit finalises.

Routes:

  • GET /me/projects/{project_id}/daily-logs — list by date range
  • POST /me/projects/{project_id}/daily-logs — create draft (auto-populates)
  • GET /me/projects/{project_id}/daily-logs/{log_id} — fetch
  • PATCH /me/projects/{project_id}/daily-logs/{log_id} — update manual fields
  • POST /me/projects/{project_id}/daily-logs/{log_id}/submit — finalise
  • GET /me/daily-logs/missing — projects without today's log

P1-T3: DailyLog voice input integration 8 hours

Extend the voice parsing service to extract daily log data from the contractor's narration. After the safety inspection is parsed from voice, remaining content (work performed, materials, delays) is extracted as daily log fields.

P1-T4: DailyLog frontend — list and detail views 20 hours

Daily log list page (by project, by date), detail/edit view with auto-populated fields shown as read-only and manual fields as editable. Clear visual distinction between "auto-captured" and "you need to add this."

P1-T5: DailyLog PDF export 8 hours

Generate a professional PDF daily log report. Format follows AIA G703/industry standard layout. Includes all auto-populated and manual fields, attached photos, and linked inspection/incident/hazard references.

P1-T6: Dashboard integration 6 hours

Add "Daily Log Status" panel to the main dashboard showing which projects have/haven't submitted today's log. Missing logs surface as amber alerts.

Phase 2: Quality Inspections & Punch Lists

Weeks 16-18 • ~74 hours • 2-3 weeks

Goal: The same inspection engine that handles safety inspections also handles quality inspections and punch lists. One walk, two reports. No second app.

P2-T1: Quality inspection templates 12 hours

Quality-specific checklist templates for common construction activities: concrete placement, framing, MEP rough-in, waterproofing, roofing, drywall, painting/finishes. Templates reference building codes (IBC/IRC) instead of OSHA standards.

P2-T2: Quality inspection backend 8 hours

Extend the existing inspection service to support quality-category inspections. Same CRUD operations, same corrective action workflow, different template set.

P2-T3: Punch list workflow 14 hours

Closeout-specific inspection type: identified → assigned → corrected → verified → closed. Punch items can be created from quality inspections or standalone walkthroughs.

P2-T4: Quality inspection frontend 12 hours

Same layout as safety inspections with quality-specific templates. Tab or toggle to switch between safety and quality views.

P2-T5: Punch list frontend 16 hours

List of deficiency items with photo, location, assignee, status, and due date. Progress bar showing items by status. Bulk update capabilities.

P2-T6: Quality metrics on dashboard 8 hours

Quality deficiency rates, punch list aging, and FTQ (first-time quality) score.

P2-T7: Daily log integration 4 hours

Quality observations from the morning walk auto-populate into the daily log alongside safety observations. One walk produces safety + quality + daily log.

Phase 3: Time Tracking

Weeks 19-22 • ~114 hours • 3-4 weeks

Goal: GPS-verified clock-in/clock-out with cost code allocation, crew-based entry, and payroll export. Connected to safety data for fatigue analysis. Foundation for auto-derived job costing in Phase 7.

P3-T1: Time entry data model 10 hours

TimeEntry model supporting individual and crew-based clock-in/out. GPS-verified. Cost code allocation. Split entries for workers moving between cost codes within a day.

P3-T2: Time tracking backend service 20 hours

Auto-calculate regular vs overtime, break deduction rules, GPS verification against project geofence, duplicate detection.

P3-T3: Cost code management 6 hours

Company-configurable cost codes. Common construction defaults provided. Cost codes assigned per project.

P3-T4: GPS geofencing service 10 hours

Define project geofences (polygon or radius). Validate clock-in/clock-out location. Flag out-of-range entries for review.

P3-T5: Safety-time correlation engine 12 hours

The unique differentiator. Flag 10+ hour workers (fatigue risk). Correlate hours with incident rates. Verify certified workers on tasks. Calculate OSHA incident rates using actual hours.

P3-T6: Time tracking frontend — worker clock-in 16 hours

One-tap clock-in/out on mobile. GPS indicator. Switch cost codes without clocking out. Spanish-first interface for field workers.

P3-T7: Time tracking frontend — crew entry 12 hours

Foreman can enter time for entire crew. Select workers from project roster, set common time and cost code, adjust individuals as needed.

P3-T8: Timesheet management frontend 16 hours

Weekly timesheet view. Per-worker, per-project, per-cost-code breakdown. Approve/reject workflow. Export to CSV for payroll.

P3-T9: Dashboard integration 8 hours

Total hours this week, labour cost by project, overtime alerts, fatigue risk flags, timesheet submission status.

P3-T10: Daily log integration 4 hours

Auto-populate daily log manpower section from time tracking data.

Phase 4: Expanded Sub Management

Weeks 23-25 • ~72 hours • 2-3 weeks

Goal: Automated COI tracking, insurance expiry alerts, sub performance scoring, and lien waiver management.

P4-T1: COI tracking and AI parsing 16 hours

Upload insurance certificates. AI/OCR extracts carrier, policy number, coverage limits, additional insured status, effective/expiry dates. Validate against GC requirements.

P4-T2: Insurance expiry monitoring 10 hours

Automated alerts at 30/14/7 days before expiry. Email to sub. Escalation to GC if not renewed.

P4-T3: Sub performance scoring 12 hours

Score from existing data: inspection pass rates, incident frequency, toolbox talk completion, corrective action closure time, training currency.

P4-T4: Lien waiver management 10 hours

Track conditional/unconditional lien waivers per payment application. Generate waiver forms.

P4-T5: Sub management frontend 24 hours

Enhanced sub compliance dashboard. Per-sub cards with insurance status, safety score, training currency, inspection activity, lien waiver status.

Phase 6: Basic Scheduling

Weeks 30-32 • ~70 hours • 2-3 weeks

Task entity with name, project, trade/crew assignment, planned dates, status, dependencies (simple finish-to-start).

P6-T2: Safety-schedule conflict detection 14 hours

The differentiator. Check cert validity through task dates, equipment maintenance due, open corrective actions, recent incidents in work area.

P6-T3: Schedule API 10 hours

Full schedule, 2-4 week lookahead, task CRUD, conflict detection, cross-project today view.

P6-T4: Schedule frontend — lookahead board 24 hours

Mobile-first weekly lookahead. Column per day, cards per task, drag-and-drop reschedule, conflict indicators, swipe to complete.

P6-T5: Schedule-daily log integration 6 hours

Planned vs actual. Auto-generate variance notes where planned tasks were not completed.

P6-T6: Morning brief integration 6 hours

Today's scheduled tasks, flagged conflicts, and key milestones approaching.

Cross-Cutting Concerns

Conversational-First, Voice Everywhere

Every feature is accessed through the conversational interface (chat and voice). The contractor creates daily logs, reviews quality observations, checks schedules, creates estimates, asks questions about job costs, and manages their projects by talking to Kerf. Dashboard and form views exist for review and bulk operations, but conversation is always the primary path.

Conversation Recording

Every interaction with Kerf — chat or voice — must be recorded and linked in the knowledge graph:

  • Full transcript stored in object storage (90-day retention)
  • Vector embedding stored as a property on the Conversation node in Neo4j
  • Extracted decisions stored as Decision nodes with description, reasoning, and links to affected entities
  • Extracted insights stored as Insight nodes with content, confidence, and applicability tags
  • All entities referenced during the conversation linked via graph relationships
  • Cross-cutting requirement: must be built alongside Phase 1 or earlier

Document Upload Pipeline

Uploaded documents (plans, specs, contracts, COIs, permits) must be processed and connected to the graph:

  • Raw files stored in object storage (permanent retention)
  • AI extraction produces structured data as graph entities
  • Text content chunked and embedded as DocumentChunk nodes with vector properties
  • All extracted data linked to the project and related entities
  • Prerequisite for Phase 7 but document upload capability should be available from Phase 1

Data Integration Points

  • Daily logs auto-populate from safety + time + equipment data
  • Quality inspections use the same engine as safety inspections
  • Time tracking connects to worker records and project assignments
  • Schedule connects to certifications, equipment, and incidents
  • Sub management uses safety data for performance scoring
  • Morning brief incorporates schedule, time, quality, and sub data
  • Estimates become tasks become cost tracking lines (unified lifecycle)
  • Job costing derives from time entries + daily logs (no manual data entry)
  • Change orders assemble evidence from daily logs + photos + time entries

Mobile-First Design

Every expansion feature must work on mobile first. The contractor is on a jobsite, not at a desk. Time tracking clock-in, daily log review, punch list walkthrough, schedule lookahead — all must be designed for a phone in direct sunlight with dirty gloves.

Testing Strategy

  • Backend: pytest with Neo4j test instance, same pattern as Phase 1
  • Frontend: Vitest + Testing Library, same pattern as Phase 1
  • E2E: Playwright journeys for each new workflow
  • Integration: IS-X scenarios for cross-feature data flow (daily log auto-population, safety-schedule conflicts, fatigue analysis)

This plan is a living document. Effort estimates are based on the existing codebase patterns and assume one developer. Actual timelines may vary. The priority order is firm — daily logs before quality before time tracking — but individual tasks within each phase may be reordered based on discovery during implementation.
