1. Principles

  1. Operator controls state transitions (Principle #15). Dashboard surfaces urgent items; tabs and quick-actions never auto-fire.
  2. Conversation is primary (§3.13, §3.15). Every tap has a voice equivalent. Every card is a shortcut for a chat intent.
  3. Dashboards all the way down. Meta-tabs open to their own dashboards, not sub-tabs. Drill happens by clicking entities.
  4. Canvas on demand (§3.15). Open questions render a canvas in a middle pane.
  5. Graph-native everything. Every number traces to the graph.

5. Meta-tab dashboards · state-aware

Dashboard

The state-aware primary surface. Content reshapes per state; frame is consistent.

  • LEAD — next-action card · Peachtree reference · client card · quick-actions
  • QUOTED — the quote document (7 numbered sections: WorkItems · Assumptions · Exclusions · Payment Schedule · Warranty · Retention · Sources)
  • ACCEPTED — contract-reference strip · planned-start tiles · "Before we start" checklist · recent activity
  • ACTIVE — "⚡ Needs your call" panel · progress by WorkItem · today · quick actions
  • PC — PC date / final value / margin / DLP tiles · close-out status · DLP defects · insights harvested
  • CLOSED — revive card · final summary · collapsed timeline

Work

Scope · WorkItems · Variations · Schedule · Daily logs · Timeline. Dashboard surfaces state-relevant cuts of each.

Site (renamed from Field)

Safety · Quality · Crew · Equipment. Rationale: one daily site walk produces all four surfaces' data. Quality belongs with Safety (same engine, same rhythm), not with Work.

Money

Invoices · Payments · Cost tracking · Margin · Variations (financial view) · QuickBooks sync · Retention.

Contract

ContractVersion history · Assumptions · Exclusions · Warranty · Retention terms · Jurisdiction · Signed documents · Disputes (with prominent chip when open).

Docs

Plans · Specs · Contracts · Permits · COIs · Photos · Reports · Emails.

People

One tab with role chips distinguishing Client · GC · Subs · Inspectors · Own crew. When Sarah works AS a sub for Peterson GC, Peterson's compliance requirements (prequal, COI, training) live on the Peterson contact drill inside People.

10. Home screen (follow-on design pass)

Project screen is project-scoped. Kerf also needs a Home screen at the app level — the default landing after login.

Home surfaces: morning brief (cross-project), active projects summary (card per project), "Needs your call" aggregate, pipeline, cash overview, canvas entry for cross-project questions.

Home is NOT a replacement for per-project detail. It summarises and routes. The Project screen is where the work lives. Its own design pass — not in this document.

1 Layout shell

  • Icon rail: exactly 48px wide, --ink-1 background, 1px hairline right border. Logo tile 28×28 at top, machine yellow, white "K". Rail buttons 32×32, 6px gap. Active button has --ink-3 bg + machine-yellow icon.
  • Agent panel: 300–320px desktop width, --ink-1 background. Three regions: header, body (flex-1 overflow-y-auto), input. On mobile: floating action button expanding to fullscreen chat.
  • Project canvas: --ink-0 background. Canvas body padding 22px.

2 Top header

  • Height ≈ 54px (14px padding top/bottom + content), --ink-2 bg, 1px hairline bottom border.
  • Logo tile: 22×22px, machine yellow, white "K" — same treatment as icon-rail logo but smaller.
  • Project title: 15px semibold. Left of the project code.
  • Project code: mono, 11px, 0.06em, machine-yellow text on --machine-wash bg. Reads like a stamped part number.
  • Dot separators (·) mono 11px --fg-dim.
  • Client name: mono, 11px uppercase, --fg-muted.
  • State chip: full-strength per-state styling (§4).
  • Right cluster: 2-line total (kicker above value, right-aligned) + CTA button.

3 Tab bar

  • Background --ink-1. Horizontal padding 22px.
  • Tab padding: 12px top, 14px horizontal, 11px bottom.
  • Tab typography: mono, 11px, 0.06em, uppercase, 500.
  • Active tab: text --machine, background --ink-0 (reads as cut into content below), 2px machine-yellow bottom border.
  • Disabled tab: text --fg-dim, opacity 0.4. Used when state doesn't have that data yet.
  • Count badges: small inline pill at 10px mono, --ink-3 bg on inactive, --machine-wash on active.
  • Do not add icons inside tabs — the tab name already starts with a glyph.

5 Overlay chips

  • PAUSED chip always prefixed ⏸, warn-yellow bg/text. Also renders a body band above the meta-tab content with reason, paused-at, who, expected resume, cure deadline. Don't collapse this into the chip alone.
  • DLP_OPEN, RETENTION_HELD — muted chips.
  • DISPUTE_OPEN — loud red. Count also appears on Contract tab.

6 Metric tiles

  • Grid: 4 equal columns, 12px gap.
  • Primary tile (always first): --machine bg, --ink-0 text. The "most important metric for this state".
  • Tile anatomy: kicker (10px mono 0.12em uppercase) · value (22px mono 600) · sub (optional 11px mono).
  • Variance colouring: sub becomes --fail when over-budget, --pass when favourable.

7 Section cards

  • --ink-2 bg, 1px hairline border, 4px radius, 14px margin-bottom.
  • accent variant: border --machine. warn variant: border --warn.
  • Section head: num · title · chip-mini · spacer · action.
  • Glyph vocabulary: ⚒ Work/Scope · ⬒ Variations · ⚠ Assumptions · ⊘ Exclusions · ⚭ Payment · ❑ Warranty · ▤ Retention · ⚙ Sources/Items · ◉ Dashboard/Today · ◎ Conversation/Timeline · ⚑ Insight · ⚡ Needs-your-call · ⛏ Labour · ✧ Methodology · ⚬ Provenance · ⟲ Revive. Implementers must not omit.

8 WorkItem table

  • Header row: mono 10px 0.06em uppercase --fg-muted on --ink-1 bg.
  • Sell column always in --machine colour, 500 weight. Visual anchor for the money decision.
  • Margin column always in --pass.
  • Sub-chips on description cells: default (--ink-3 + muted) for refs like "← Peachtree"; insight (--machine-wash + machine) for "⚑ +15% insight".
  • Selected row: --machine-wash bg, 3px machine-yellow left-border, padding-left reduced by 3 — row depth doesn't change.

9 Progress bars

  • Track 4px tall, --ink-3, 2px radius.
  • Fill default --machine. At 100% flips to --pass (green) — visual reward for completion.
  • Percentage label mono 10px, never drops the % sign.
  • Min container width 110px.

10 Chat pane — the event icon requirement

This is the area specifically flagged as easy to lose.

Event markers (distinct from messages)

  • Dashed border (not solid) — 1px dashed --hairline.
  • Background --ink-2 (same as Kerf bubbles, dashed border separates them as a class).
  • Padding 8×10px.
  • Typography mono 10.5px 0.04em --fg-muted — explicitly smaller than messages.
  • No avatar, no "who" line — events are system-generated.
  • Always prefixed with ◈ (diamond glyph). Present on every event.
  • Strong text inside events uses --machine for entity names, sums, keywords.

Implementation guard: events are their own component, not a styled variant of ChatBubble. Build ChatEvent separately so the visual contract can't regress.

15 Methodology cascade

  • Three levels with level-specific left border: Project --machine, Package --fg-muted, Item --fg-dim.
  • Keys in 2-col grid. Override key: border --machine, bg --machine-wash, value --machine, key prefixed ⚑.
  • Effective map block: --ink-3 bg + --machine-wash border. Key · value · origin columns.

16 Lineage equation

  • Background --ink-3. Mono typography, 12px.
  • Node variants:
    • Default: --ink-4 bg, --fg, hairline border (e.g. "12 holes", "16.0 hrs")
    • Base: --pass-bg bg, --pass (known-good rate, e.g. "1.33 hr/hole")
    • Insight: --machine-wash bg, --machine (the adjustment, e.g. "low-ceiling")
    • Final: --machine bg solid, --ink-0, 600 weight (the computed answer)
  • Arrows --fg-dim text. Can be →, ×, =, +15%.

17 Provenance tree

  • ASCII tree characters ├─ │ └─ in --fg-dim.
  • Mono 12px, 1.9 line-height for airy readability.
  • Indent 16px per level.
  • Node values in --machine or --pass by semantic.
  • Inline source tags embedded in the descriptive line.

1 Project state machine

Existing Project.state has lead | quoted | active | completed | closed | lost. New enum: LEAD | QUOTED | ACCEPTED | ACTIVE | PRACTICAL_COMPLETION | CLOSED. New properties: closed_reason (typed), state_at_closure.

Existing status field (on_hold / suspended) → removed, replaced by PAUSED overlay entity.

CREATE CONSTRAINT constraint_project_state_valid IF NOT EXISTS
  FOR (p:Project) REQUIRE p.state IN
    ['LEAD','QUOTED','ACCEPTED','ACTIVE','PRACTICAL_COMPLETION','CLOSED'];

CREATE INDEX index_project_state IF NOT EXISTS FOR (p:Project) ON (p.state);
CREATE INDEX index_project_closed_reason IF NOT EXISTS FOR (p:Project) ON (p.closed_reason);

Migration in the source doc maps legacy states. Lossy: existing "completed" projects map to PRACTICAL_COMPLETION; explicit user action needed to CLOSED.

2 ContractVersion

New entity. Immutable snapshot captured at QUOTED → ACCEPTED.

Properties: id · version · project_id · sum_cents · doc_hash (sha256 of accepted doc) · parties · dates · payment_terms · retention_pct · jurisdiction · governing_law · accepted_by · accepted_at · accepted_method.

Relationship: (Project)-[:HAS_CONTRACT_VERSION]->(ContractVersion).

3 Variation

New first-class entity with its own lifecycle. Numbered addenda to ContractVersion v1 — amendment-in-place, never new ContractVersions.

State enum: Draft | Issued | Approved | Rejected | Superseded.

Key property: number — sequential per contract, gap-free. Validation query catches gaps/duplicates.

Relationships:

  • (ContractVersion)-[:HAS_VARIATION]->(Variation)
  • (Variation)-[:HAS_LINE_ITEM]->(WorkItem) — zero or more, new or modified items
  • (Variation)-[:SUPERSEDES]->(Variation) — when revised (V1.1 supersedes V1)
  • (Variation)-[:EVIDENCE_FROM]->(DailyLog|Photo|TimeEntry|Conversation)

Effective contract sum is never stored — always computed as sum(ContractVersion.sum_cents) + sum(Approved Variations.sum_delta_cents).

4 Dispute

New first-class entity with its own lifecycle. Multiple disputes per Project supported.

State enum: Raised | Negotiating | Adjudication | Mediation | Arbitration | Resolved.

Relationships: DISPUTE_OVER, HAS_EVIDENCE, HAS_ADJUDICATOR, RESOLVED_BY.

DISPUTE_OPEN overlay is derived — true when any linked Dispute is un-resolved.

5 Revival event

New entity capturing state reversal.

Properties: id · revived_at · revived_by · prior_closed_reason · state_before_revival (always "CLOSED") · state_after_revival (any active state) · note.

Relationship: (Project)-[:HAS_REVIVAL]->(Revival).

6 Overlay entities

Overlays are first-class entities, not relationship properties. Enables clean history, audit, re-activation.

  • PauseRecord — pause_type (on_hold / suspended) · reason · paused_at/by · expected_resume_at · cure_deadline · resumed_at/by.
  • DefectsLiabilityPeriod — source (contract_clause / statutory) · dlp_start · duration · end · closed_at.
  • RetentionHolding — retention_pct · held_in_trust · trust_account_ref · amount_cents · scheduled_releases · released_at.

7 WorkItem state enum addition

New property: state — not_started | in_progress | complete | de_scoped. Existing percent_complete 0–100 stays.

PC precondition: all in-scope WorkItems at complete or de_scoped. Validation query enforces.

8 Quote versioning

New. First-class versionable documents.

State enum: Draft | Sent | Accepted | Rejected | Superseded | Withdrawn.

Relationships: HAS_QUOTE (from Project), SUPERSEDES (quote-to-quote), ACCEPTED_INTO (quote-to-ContractVersion on acceptance).

9 DefectItem

Used for PC punch lists and DLP defects. Category (cosmetic / functional / safety), status (open / resolved / dismissed), origin (punch_list_at_pc / dlp_defect), responsible_party, resolution metadata.

10 Source provenance on Labour + Item

Extends existing Domain 16/17 nodes.

New Labour properties: productivity_source · rate_source · rate_source_id · productivity_source_id · applied_insight_ids · base_hours · applied_adjustments.

New Item properties: price_source · price_source_id · applied_insight_ids.

Indexes on rate_source, productivity_source, price_source.

11 Email tracking

Supports §3.14 Email as Client Channel.

  • EmailThread — project_id · subject · initiated_by · provider (gmail / outlook) · provider_thread_id.
  • EmailMessage — direction (inbound / outbound) · state · from · to · cc · body · sent_at · received_at · drafted_by_agent · approved_by.
  • Validation: outbound sent messages must have approved_by.

The state machine

Six states. Amendment-in-place variations. First-class Dispute entity. PauseRecord overlays. Revive by state reversal with audit. Every transition operator-controlled per Principle #15.

PC pivot transition

Biggest transition operationally. At PC: risk shifts to owner · insurance shifts · LDs stop · retention half-release where applicable · DLP clock starts · contract sum frozen except outstanding claims.

  • Trigger: operator declares done. One hard precondition — all in-scope WIs at complete or de_scoped.
  • Snapshot: PC date · scope-at-PC (v1 + approved variations) · punch list (from operator capture) · DLP terms where applicable.
  • Nudges (not gates): open variations · missing final inspection where permitted · outstanding invoice balance.
  • Client acknowledgment is an event, not a gate. Kerf sends PC certificate via email + magic-link acknowledge. Absence is a gap in the audit trail, not a blocker.

The Variation lifecycle

Draft → Issued → Approved
           ├→ Rejected
           └→ Superseded     (when operator revises after client pushback)
  • Draft: operator composing. Not visible to client.
  • Issued: sent via email + magic link. Events inside (not state changes): client viewed, asked question, requested revision.
  • Approved: client signed. Immutable. Adjusts effective contract sum. Becomes billable.
  • Rejected / Superseded: immutable terminal / intermediate.

Variations are numbered addenda (V1, V2, V3...) — amendment-in-place, not new ContractVersions. Gap-free sequence. Matches research #3 §1.2.

Bundle vs split is operator's choice at draft time. Kate's three requests → one Variation with three line items (bundle, all-or-nothing approve) or three separate Variations (split, per-item approve/reject).

Dispute lifecycle

Raised → Negotiating → [Adjudication | Mediation | Arbitration] → Resolved
                                                                      │
                                                                      └→ (possible further escalation)

First-class entity. Multiple disputes per Project supported. Does not change Project state — Project stays ACTIVE while DISPUTE_OPEN. Work continues unless a party also invokes PAUSED (suspension).

Revive

State reversal, operator-controlled. Kerf's nudges calibrated by exit reason:

FromToNudge
CLOSED[lead_lost]LEADSoft — "you closed this 6 months ago; continue?"
CLOSED[abandoned]prior stateFirmer — "abandoned mid-job; contract validity may have lapsed"
CLOSED[terminated_*]prior stateLoud legal caution — "formally terminated; consider new Project"
CLOSED[completed]—No revive; new scope = new Project linked to this one

Revival captures immutable event: revived_at · revived_by · prior_closed_reason · state_before_revival · state_after_revival · note. State history reads cleanly — closure records never deleted, stay as historical facts.
