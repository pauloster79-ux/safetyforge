Work Item UI Design

From conversation to structured work · Scales from solo to 100-person crew

The Core Idea

A WorkItem is the atomic unit of deliverable work. It has a description, optional quantity, labour entries, item entries (materials, equipment, subs — anything that isn't labour), and a lifecycle state. The user creates them by talking — the agent extracts structured work from natural language and presents them as editable cards for review.

The card IS the work item. It appears in chat as a draft. The user can confirm it (one tap), edit any field (tap the value), or dismiss it. On desktop, the same card appears in the canvas table view. There's no separate "create form" — the conversation IS the form.

Work Package

A WorkPackage is an optional grouping. It emerges naturally — when Sarah says "reception area" the agent creates a package. Or the user can group items later. Solo contractors like Jake never see packages — their items are flat on the project.

Every value with a dashed underline is tappable/clickable to edit. Tap "6 ea" → inline number input. Tap "$3.40 /ea" → edit the unit price. Tap a Category or Methodology row → opens the picker or cascade view. Tap any ↳ source line → opens the Source Detail panel (see Source Detail tab).
Category is always shown. It's required classification — every WorkItem belongs somewhere in the canonical taxonomy (MasterFormat for US/CA, NRM 2 for UK/IE, NATSPEC for AU/NZ). Methodology only appears when something's been set — either at Project, Package, or this item. Jake's jobs usually don't need explicit methodology; Sarah's commercial work usually does.

What's happening in this view

  • Segmented control (Quote · Active · Closeout) top-right of scope table — user can manually override the lifecycle view; default tracks project.state.
  • Header cards now answer "where am I?" Contract is the baseline. Spent & Forecast shows the live trajectory. Variance is the money-leaking flag. Change Orders is the scope-reconciliation queue.
  • Per-row Variance cell shows the dollar delta plus its type badge — Customer (orange) is billable via CO; Builder (red) hits margin; Weather (blue) is EOT/force-majeure.
  • Expanded row shows TimeEntries rolled up against each Labour row ("Logged: 9.0 hrs · Remaining est: 5.0 hrs") and receipts rolled up against each Item row. The source lines (↳) are still there under the baseline rate/price — tap to drill to the same Source Detail panels shown in that tab.
  • Pending Customer Variance appears inline where it was detected — with one-click "Generate Change Order" action. This is the CoConstruct-style auto-CO flow we borrowed.
  • Methodology is shown under the scope-tracking header as a project-level chip ("Method · Kitchen remodel · retain plumbing layout · conventional finishes"). No per-item methodology overrides on this simple job, so no chips on individual rows.
  • Footer has two rows now: a dark "Forecast at complete" roll-up (est → forecast for Labour, Items, with total variance) and the machine-yellow "Margin: planned 20.0% · forecast 13.9%" — the gut-check number.
Why the same table works for three lifecycle stages. Quote view has columns Margin / Sell. Active view replaces those with est→act cells, % done, and variance. Closeout view (coming in a future preview tab) replaces those with Invoiced / Paid / Outstanding. The row is the stable spine; the columns are the lens. This mirrors the Procore / Autodesk / Buildertrend pattern but uses our own variance typing and source provenance.

Source Detail — How users see provenance

Tap any source line (↳) on a card · panel opens with the evidence chain · contractor can accept, override, or investigate
This is Kerf's biggest differentiation. No competitor we surveyed shows contractors where a number came from. Every rate, price, productivity adjustment, methodology decision, and category classification can be drilled into. The source panel shows the authoritative node in the graph, the evidence chain, confidence signals, and quick actions to override.

The interaction

On the card or grid row, every numeric value has a muted source line underneath (prefixed with ↳). Tap / click the source line — a source panel opens. On mobile, it's a bottom sheet. On desktop, it's a side panel that slides in from the right (using the existing SourcesPanel component in ContractTab.tsx). Close the panel to return to the card.

Interaction pattern

Entry points

  • ↳ source line under any numeric value on the card → opens that value's source panel
  • Methodology row on the card → opens the cascade panel
  • Category row on the card → opens the category picker (search-first); tap the info icon → opens the category source panel
  • Insight chip (purple) on the labour source line → opens the productivity source panel pre-scrolled to the Insight section
  • Grid row context menu (desktop) → "View sources" option opens all five panels stacked

Mobile behaviour

  • Bottom sheet slides up, 80% screen height. Tap outside or swipe down to dismiss.
  • Headers sticky so the value stays visible while scrolling evidence.
  • Action buttons fixed at the bottom of the sheet.

Desktop behaviour

  • Side panel slides in from the right, ~400px wide. Uses the existing SourcesPanel component shell.
  • Panel is dismissible by clicking outside, pressing Esc, or clicking the × in the panel header.
  • Multiple panels can stack (e.g., opening productivity while labour rate is already open pushes labour rate to the left or replaces it — design decision TBD).

Data flow

  • Each WorkItem's Labour node carries rate_source_id, productivity_source_id, applied_insight_ids.
  • Each Item node carries price_source_id, price_source_type.
  • Each Methodology carries parent references and an approach map with per-key overrides.
  • Each Category is either a Canonical node (shared) or an Extension (company-scoped under a Canonical parent).
  • The panel resolves the references in one query and renders the evidence tree.
The "Override" button closes the loop. If the contractor disagrees with the system's choice, they can override at item-level (one-off), at package-level (for this project), or globally (for all future projects). Each override is itself captured with provenance, so the system can learn from the corrections and — over time — the Insight confidence improves without anyone authoring "templates" from scratch.
