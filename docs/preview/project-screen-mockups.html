Project Screen · Structural Options · 2026-04-18

How the Project screen reshapes across states

The question: how does the Project screen look in ACTIVE vs QUOTED — do we have tabs, and how is all the functionality structured? Claude Design already produced a beautiful QUOTED screen (shown as Option A.1 below). This document shows three structural options for the ACTIVE state alongside it, using Jake / Maple Ridge data for continuity, so we can compare them directly. Every mockup uses the Site Board design system exactly as defined in frontend/src/index.css — same palette, same typography, same component language.

Each option carries the same header, agent panel, and general shell. What differs is how the body is organised — sectioned scroll vs. tabbed dashboard vs. state-aware primary canvas + tabs.

REFERENCE · A.1 · QUOTED state · sectioned scroll

The quote IS the document

This is the shape Claude Design produced. Seven numbered sections compose the proposal. The contractor's task is compose + review + send — a document-oriented task. Scrolling top-to-bottom reads like reading the quote.

✓ Right fit for QUOTED — document oriented ✗ Wrong fit for ACTIVE — contractor isn't composing, they're monitoring

Observation The QUOTED canvas scrolls as a single document. That's right for QUOTED — the contractor is composing and reviewing. Sections 02 through 07 are the commercial structure of the proposal; they're all visible at once because the task is read the whole quote before sending. In ACTIVE, the task changes. The contractor isn't re-reading the assumptions daily — they're logging time, checking progress, drafting variations. That's where a different canvas structure wants to land.
Option A.2 · ACTIVE state · same sectioned scroll pattern

ACTIVE as a long scroll — same shape, different content

If we keep the QUOTED shape and just swap the content per state, ACTIVE looks like this. Contract reference (assumptions / exclusions / warranty / retention) is compressed into a single reference strip at the top. Execution content — WorkItems with progress, variations register, timeline, upcoming — fills the numbered sections.

✓ Consistent visual shape across all states ✗ Long scroll to reach "log time now" — wrong for a dashboard task ✗ ACTIVE has many more live elements than QUOTED — harder to keep scanable

Verdict Option A.2 works, but the screen feels crowded — contract reference compressed upward, execution data sprawling downward. The QUICK ACTION Sarah most needs (approve V1, log time, respond to a question) requires scrolling through the scan. Fine for weekly review, not great for a glance from the truck.
Option B · ACTIVE state · explicit tabs

ACTIVE as a tabbed workspace

Replace the long scroll with named tabs. Landing tab = Overview (metrics, today, quick actions). Other tabs own their own deep views: Scope, Variations, Timeline, Money, Docs, People. Contract reference is always one tab click away. Used consistently across states: same tab bar, but the Overview content reshapes based on state.

✓ One-tap to the task you care about ✓ Consistent navigation across all states ~ Tabs add chrome — a chunk of vertical space gone ~ Overview design still matters most; tabs alone don't solve anything

Verdict Tabs give Sarah one-tap access to the task she cares about and reclaim the vertical space. The trade is that some information that felt valuable at-a-glance in Option A.2 (variations register, payment schedule) now lives a tab away. Whether that's right depends on frequency of use. Approving a draft variation is a one-a-week action — it's fine behind a tab.
Option C · ACTIVE state · state-aware primary + secondary tabs (recommended)

A state-shaped primary view, with tabs for drilling

The landing content reshapes per state — QUOTED lands on the quote document, ACTIVE lands on an execution dashboard, PC lands on close-out — while the secondary tabs (Scope · Variations · Timeline · Contract · Docs · People) stay consistent. The "Overview" dissolves into the state — no tab needed for it, the main canvas IS it. More bespoke per state, but less generic tab-chrome.

✓ Each state lands on exactly the right primary view ✓ Tabs carry the always-there concerns — no redundant "Overview" tab ✓ Feels less like admin-tool, more like product ~ More work per-state to design (that's the point)
