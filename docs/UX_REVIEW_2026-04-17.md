# Kerf UX Review

*Date: 2026-04-17*
*Reviewer: Claude (session handoff doc)*
*Scope: Full-product UX review — flow, screen layout, componentry, visual language — measured against `docs/PRODUCT_VISION.md` and the three personas (Marco, Sarah, Jake).*

---

## Top-line verdict

Kerf has the right architectural skeleton for what the vision describes — a three-pane conversational shell with chat as a persistent surface, a voice inspection route, a rich domain model exposed as tool-result cards, and tokens that signal "construction" rather than "SaaS template." But three things are holding it back from feeling like a funded product:

1. **The visual language reads as an internal admin tool, not a product.** IBM Plex + 3 px radii + micro-typography (10–12 px uppercase mono labels) + near-zero shadows/motion gives a clipboard/bulletin-board aesthetic. That's a defensible stylistic direction but it's currently executed at 3–4/10 polish, not 8/10.
2. **The conversational primacy is architectural, not behavioural.** Chat is always on-screen, but the actual field workflows (inspections, hazards, adding a worker) are still multi-step forms. Voice lives behind one dedicated route rather than being ambient. The Morning Brief waits to be asked, not waiting on open.
3. **The field persona (Marco at 5:30 AM on a phone) is underserved.** Desktop gets the three-pane treatment; mobile gets a collapsed version of it. There's no evidence of offline queueing, no voice-note on the hazard report, no walking-estimate flow, no "start a site walk" primary affordance.

Below is the detail and the priority list.

---

## What's working (don't throw this away)

- **Three-pane shell (rail + chat + canvas)** at `frontend/src/components/shell/AppShell.tsx`. This is a thoughtful, differentiated IA — chat is not a widget, it's a pane. Keep this.
- **Tool-result card system** (30+ domain cards in `frontend/src/components/cards/`) — the pattern of "agent calls a tool → rich card appears in chat → clicking opens the canvas" is exactly right for a graph-native conversational product.
- **Voice Inspection full-screen mode** at `frontend/src/components/voice-inspection/VoiceInspectionPage.tsx` — the orb + transcript + interruption model is the correct metaphor.
- **Token discipline** in `frontend/src/index.css` — single brand color (`#F5B800` machine yellow), semantic `pass/fail/warn` tokens with translucent bg variants, dark mode wired through. The foundation is sound.
- **Warm neutral palette** (`#f4f5f3` / `#0d0e0c`) instead of the cold gray default — distinctive, trade-appropriate.

## Where reality diverges from vision

### 1. Visual language: industrial, but under-crafted

The product is clearly *trying* for a "blueprint / engineering clipboard" aesthetic — uppercase mono labels, tight 3 px radii, `FormSection` boxes with hairline borders. That's a legitimate direction (think Linear-meets-Revit). But the execution stops at "functional":

- **Type scale is missing.** Inline `text-[10px]`, `text-[11px]`, `text-[12px]` everywhere. No defined `display / h1 / h2 / body / caption` tokens. Headings are just component titles. This is the single biggest reason the UI reads as admin-panel: no hierarchy, nothing earns attention.
- **Zero depth.** Almost no shadows, no layering, no motion. shadcn defaults have been flattened further. Cards sit flat on flat. On modern devices (OLED phones, retina laptops), this looks unfinished rather than minimal.
- **Brand color is underused.** A bright confident yellow exists at `--primary` and sits largely unused outside the LogoMark and a few badge states. There's no brand moment on any screen.
- **Iconography is generic.** 103 Lucide imports; no custom mark besides the checkmark-in-yellow-square logo. No iconographic language that says "construction" (hard hat, clipboard, plumb line, scaffold, drawing tools).
- **Typography choice.** IBM Plex is a safe, engineering-adjacent pick. It's not wrong, but it's also not distinctive. Procore uses Proxima Nova; Linear uses Inter with care; modern SaaS uses Geist. Plex alone isn't carrying brand weight — if Plex is the choice, it needs opinionated sizing/weights to feel intentional rather than defaulted.
- **Landing page** (`frontend/src/components/landing/LandingPage.tsx`) uses the same `FormSection` boxes as the app — which is clever (consistency) but flat for marketing. First impression is "internal portal," not "product a contractor would pay for."

### 2. Conversational is the surface, not the workflow

The shell positions chat correctly, but once you click into any canvas page, the conversational promise breaks:

- **Inspection creation** is a multi-step wizard (`frontend/src/components/inspections/InspectionCreatePage.tsx:53`): pick type → fill header → pass/fail per item → summary notes. That's 20+ taps for a single walk. The Voice Inspection route is parallel to this, not the primary path. Marco will use whichever is faster; right now, neither feels like "three taps."
- **Hazard report** (`frontend/src/components/hazards/HazardReportPage.tsx:40`) has photo + text description but **no voice note field**. The vision's flagship moment — Carlos snaps a photo and speaks in Spanish — doesn't exist yet.
- **Worker create** is an 8-field form. No "say the worker's name and I'll fill in the rest" flow.
- **Estimate creation** is chat-only — there's no walking-estimate mode (mic button + live WorkItem cards). The `EstimateSummaryCard` is read-only.
- **Voice is locked to one route.** `/projects/:projectId/inspect/voice` is the only place the mic surfaces. The main `ChatPane` input is text-only. Mic should be persistent, one tap away, from any screen.

### 3. Morning Brief waits to be asked

`DashboardPage` calls `useMorningBrief(firstProjectId)` but doesn't automatically push the brief into chat on open. The vision says *"Marco opens Kerf at 5:30 AM. The morning brief is waiting."* Today, he opens Kerf and sees a status strip with project counts and TRIR — a metrics dashboard, not a brief. The brief is *available*, not *waiting*.

Same pattern repeats elsewhere: daily logs auto-populate only when the chat asks; cert expiries are a number in a strip, not a prompt that arrives. The product is reactive where it's supposed to be proactive.

### 4. Information architecture: 12 rail icons, siloed canvas

The IconRail has 12 destinations: Chat, Projects, Schedule, Daily Logs, Workers, Equipment, Safety, Documents, Reports, Queries, Knowledge, Settings. That's a traditional module menu. The vision is *"one tool, not six"* — but the IA currently lists six+ modules. Two problems:

- **Marco doesn't think in modules.** He thinks "the pour tomorrow," "Javier's cert," "Peterson GC." The rail organises by data type (Workers, Equipment, Documents) rather than by work (Today, This Week, Open Items). A project-first or day-first IA would match the mental model better.
- **Graph connectivity is invisible.** The canvas pages are tabbed detail views (ProjectDetailPage has Safety/Team/Work/Contract tabs). None of them surface the traversable graph: "this project needs OSHA-30 for scaffold work → 2 of 5 assigned workers hold it → Javier's expires Wednesday → Friday's pour is at risk." That causal chain is the product's competitive moat. It should be a first-class UI pattern, not something you only access by asking in chat.

### 5. Craft details that add up

- **Empty states are silent.** `InspectionListPage` returns `null` when there's no data. A contractor opening the app on day two sees blank space. These are missed onboarding opportunities *and* brand moments.
- **Loading is a spinner.** No skeletons, no progressive reveal, no shimmer. Chat streaming arrives into an empty bubble.
- **No microcopy voice.** Labels read like field names (`Inspector Name`, `Weather`, `Date`). A product voice would say "Who's doing this walk?" or "What's it like out there?" The vision has a tone — Kerf should have a voice.
- **Cards are a patchwork.** 30+ bespoke card components with no shared skeleton: header pattern, citation footer, action affordance, density rules. They work but don't feel like they came from the same design team.
- **No DESIGN_SYSTEM.md.** Not in `docs/`. For a product that wants to feel premium, this is the foundation missing.
- **Print styles exist, motion doesn't.** That's backwards for a conversational-first, field-first app.

### 6. Persona fit against the vision stories

Tested against the two hero stories in `PRODUCT_VISION.md` section 7:

| Moment from vision | What Marco/Sarah would hit today |
|---|---|
| 5:30 AM: brief is waiting | Dashboard with KPI strip, brief available if asked |
| 6:15 AM clip phone, record 8-min site walk | Voice Inspection page exists but is separate from form-based InspectionCreatePage; no auto-generation of quality + daily log from one walk |
| 9:45 AM Carlos reports hazard in Spanish with photo + voice | No voice note on hazard form; no visible language toggle on hazard screen |
| Sarah walks the 4th Street site, dictates an estimate | No walking-estimate UI; no live WorkItem cards; no voice-to-estimate flow at all |
| "How did we do on the Peachtree job?" narrative answer | Chat + cards exist — this one is closest |

The conversational-report moment is the strongest of the five. The other four — which are the field-defining moments — all have significant gaps.

## Priority recommendations

I'd sequence this as three waves. Each has a clear "what changes for the user" outcome.

### Wave 1 — Brand & craft foundation (2–3 weeks, highest ROI for "polished" perception)

1. **Write a design system** (`docs/DESIGN_SYSTEM.md`) with: type scale (display/h1/h2/body/caption/mono-label in rem), spacing scale, radius scale, elevation scale (even if subtle), motion tokens (durations + easings), iconography rules, voice-and-tone guide. This is the document that the next designer or AI-assisted contributor reads before touching anything.
2. **Fix the type scale.** Replace inline `text-[10px]` / `text-[11px]` / `text-[12px]` with semantic classes. Establish a display size (e.g., 40/44 semi-bold) that lets screens breathe. Hierarchy alone will move the polish score from 3 to 6.
3. **Introduce one extra axis of depth.** Either a subtle elevation system (2 or 3 shadow levels, used sparingly) *or* commit to strict flat design with color-only separation (like Linear). Current state is neither.
4. **Use the yellow.** It's the distinctive asset; it's barely deployed. Yellow moments on primary CTAs, active nav states, hero card accents. Not everything yellow — but at least *something* yellow on every screen.
5. **Custom iconography for the five core domains** (Safety, Quality, Daily, Time, Subs) plus the chat/voice mic. Even 6 custom SVGs change the brand feel disproportionately.
6. **Motion language.** 150–250ms easing on canvas open/close, card reveal, chat message slide-in, orb state transitions. Three or four tokens, used consistently.
7. **Empty states with illustrations + microcopy.** Every list. "No inspections yet — start a site walk with one tap" with a mic button. These are brand moments.
8. **Rebuild the Landing Page.** It's the first artefact anyone sees and currently looks like a settings page. Use the hero moments from the vision (Marco's 5:30 AM, Sarah's site walk, Jake in the kitchen) as the narrative spine, not feature bullet lists.

### Wave 2 — Make conversational primacy actually primary (3–4 weeks)

9. **Ambient mic.** A persistent microphone button in the ChatPane input (hold-to-speak) and a dedicated mic FAB on mobile. Voice is one tap from anywhere, not behind a route.
10. **Morning Brief as app-open behaviour.** When the first screen of the day loads, the brief is the first chat bubble — already streaming. The Dashboard is *optional*, not the default.
11. **Hazard report — voice + photo only.** Replace the description textarea with a voice-note recorder. Tap, hold, speak, release, submit. Three taps.
12. **Unified site-walk flow.** Merge VoiceInspectionPage and InspectionCreatePage into one flow where voice is the default path and the form is the fallback. The vision is "one walk, three reports" — the UI should produce three cards (safety / quality / daily log) at the end, not force a module choice at the start.
13. **Walking estimate.** New flow: "Start estimate" → mic on → live WorkItem cards appear in the canvas as the contractor narrates. Cards are editable inline. This is the feature that makes Sarah's story real.
14. **Add a voice note field everywhere there's a description.** Worker notes, corrective actions, daily log narrative, change-order justification.

### Wave 3 — Graph as a visible product surface (4–6 weeks)

15. **"Today" home.** Replace the module-list rail top item with a work-oriented view: today's brief, today's schedule, open items needing action. Organised by *work*, not by *entity type*.
16. **Graph chip/breadcrumb system.** Everywhere an entity is mentioned, it becomes a chip that shows its graph neighbourhood on hover/tap — project → workers → certs → applicable regulations. This makes the moat visible. Right now the graph is the invisible engine; users should feel it.
17. **Citations as first-class UI.** Every agent recommendation should render a citation chip ("per OSHA 1926.502(d)") that expands to show the full traversal. The "we show our work" principle from the vision deserves a dedicated component.
18. **Proactive prompts.** Background agent pushes chat messages when something needs attention — cert expiring Wednesday, weather turning, sub insurance lapsing in 12 days. The app should talk first, not only respond.
19. **Offline-first pass.** Visible offline state in the chat input (queue indicator), photo queue UI, sync badge. If it's not architecturally offline yet, at minimum the UI needs to *honour* the principle visually.

### What to deprioritise

- Mobile swipe gestures, native apps, PWA polish — none of that matters until the mobile *information architecture* matches the field moments above.
- More tool cards. There are already 30+; the next one should be deleted, not added. Consolidate into a smaller, consistent card system.
- More features in the rail. The rail should probably shrink from 12 to 6, not grow.

---

## The one-sentence brief for the design team

> Kerf today is a well-architected internal tool that's waiting for a product team to show up — it has the graph, the chat, the cards, and the tokens, but no type scale, no motion, no brand moments, no voice-everywhere, no walking estimate, and no auto-delivered morning brief; fix those and it becomes the product the vision describes.
