Domain

Construction business operations — the complete lifecycle of a contractor running projects. All sizes, all trades. Data structures are scale-invariant.

1. Find & qualifyLead comes in, decide if it's worth pursuing
2. Estimate & priceFigure out cost and markup
3. Propose & winSend quote, negotiate, get agreement

The Work Model — Two Layers + Detail

Two structural layers. Everything else is detail. Project (the job) and WorkItem(s) (the work within it). WorkPackage is an optional folder. All other entities attach as detail to one of these two layers.
The spine (always present): (Company) → (Project) → (WorkItem) + (Contact) Optional grouping: (Project) → (WorkPackage) → (WorkItem) Detail on Project: Contract, DailyLog, Inspection, Invoice, Variation, Milestone... Detail on WorkItem: Items, Workers, Crew, TimeEntries, Equipment... Jake: Project → 1 WorkItem ("Kitchen electrics, £2,400") Dave: Project → WorkPackages by trade → WorkItems Sarah: Project → WorkPackages → 200 WorkItems with Items, TimeEntries, Crew
No separate Estimate entity. Work items at "quoted" status ARE the estimate. Change history via conversation memory + provenance.

WorkItem: State + Status

state (lifecycle, always set): draft → scheduled → in_progress → complete → invoiced + on_hold, cancelled status (current condition, optional, free text): "Waiting on materials", "Blocked by framing", "Ready for inspection" Null when progressing normally. Set by agent from conversation.

Vector Indexes

CREATE VECTOR INDEX message_embeddings IF NOT EXISTS FOR (m:Message) ON (m.embedding) OPTIONS { indexConfig: { `vector.dimensions`: 1536, `vector.similarity_function`: 'cosine' } } CREATE VECTOR INDEX chunk_embeddings IF NOT EXISTS FOR (c:DocumentChunk) ON (c.embedding) OPTIONS { indexConfig: { `vector.dimensions`: 1536, `vector.similarity_function`: 'cosine' } }

Sample Queries

What work remains?

MATCH (p:Project {id: $projectId})-[:HAS_WORK_ITEM|HAS_WORK_PACKAGE*1..2]->(wi:WorkItem) WHERE wi.state IN ['draft', 'scheduled', 'in_progress', 'on_hold'] RETURN wi.description, wi.state, wi.status

Semantic search over conversations

CALL db.index.vector.queryNodes('message_embeddings', 5, $queryEmbedding) YIELD node AS msg, score WHERE score > 0.75 MATCH (msg)-[:PART_OF]->(conv)-[:ABOUT_PROJECT]->(p:Project) RETURN msg.content, conv.title, score

What regulations apply?

MATCH (a:Activity {id: $activityId})-[:REGULATED_BY]->(reg:Regulation) WHERE reg.jurisdiction_code = $jCode AND reg.valid_until > date() RETURN reg.reference, reg.title, reg.source

  • Two-layer model (Project → WorkItems) — universal pattern
  • Bottom-up costing — universal
  • No separate Estimate entity — confirmed
  • Global-first naming — genuine differentiator, no competitor does this
  • Regulatory knowledge graph (Layer 3) — completely unique
  • Conversation memory — completely unique
  • Graph-native permissions — no competitor uses this

What's Unique to Kerf

The graph architecture handles every competitor gap without adding entities. Competitors need separate entities because relational databases require explicit join tables. Our graph uses relationships and traversals. The agent's ability to traverse dynamically replaces static template logic, version tracking, and billing structures.
