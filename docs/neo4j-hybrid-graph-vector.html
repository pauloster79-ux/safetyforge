Neo4j: Hybrid Knowledge Graph + Vector Storage

How a single database can store structured knowledge, conversation history, and document embeddings — enabling hybrid retrieval that combines graph traversal with semantic similarity search.

1. The Core Idea

Neo4j is a native graph database that, since version 5.11, also supports native vector indexes. This means a single database instance can serve as both:

Knowledge Graph

Entities, relationships, rules, and metadata stored as nodes and edges — queryable via Cypher with full traversal, path-finding, and pattern matching.

Vector Store

Embedding vectors stored as properties on nodes, indexed for approximate nearest-neighbour (ANN) similarity search — no separate vector database required.

This eliminates the need to synchronise two systems (a graph DB and a separate vector DB like Pinecone, Weaviate, or Qdrant). The graph structure and the vector embeddings live on the same nodes, enabling queries that combine both in a single Cypher statement.

2. What Can Be Stored

Conversations

Chat messages, agent interactions, and conversation threads can be modelled as graph structures. Each message is a node linked to its conversation, its author (human or agent), and any entities it references.

(:Conversation {id, created_at, title}) ←[:PART_OF]— (:Message {id, role, content, timestamp, embedding}) —[:SENT_BY]→ (:User | :Agent) —[:REFERENCES]→ (:Entity) —[:FOLLOWS]→ (:Message) // preserves order

Documents & Chunks

Documents are ingested, chunked, and each chunk is stored as a node with its text and embedding vector. The graph preserves the document structure (sections, pages, parent-child) and links chunks to the domain entities they mention.

(:Document {id, title, source, ingested_at}) ←[:CHUNK_OF]— (:Chunk {id, text, page, position, embedding}) —[:MENTIONS]→ (:Regulation | :Project | :Worker | ...) —[:NEXT_CHUNK]→ (:Chunk) // preserves reading order

Domain Knowledge

The same graph holds your structured domain data — regulations, projects, workers, certifications — as richly connected nodes. This is the key advantage: document chunks and conversation messages link directly to the domain entities they relate to.

3. Native Vector Indexes

Neo4j's vector index uses Hierarchical Navigable Small World (HNSW) graphs for efficient approximate nearest-neighbour search. Vectors are stored as properties on nodes and indexed separately from the graph structure.

Creating a Vector Index

// Create a vector index on Chunk nodes
CREATE VECTOR INDEX chunk_embeddings IF NOT EXISTS
FOR (c:Chunk)
ON (c.embedding)
OPTIONS {
  indexConfig: {
    `vector.dimensions`: 1536,
    `vector.similarity_function`: 'cosine'
  }
}

// Create a vector index on Message nodes
CREATE VECTOR INDEX message_embeddings IF NOT EXISTS
FOR (m:Message)
ON (m.embedding)
OPTIONS {
  indexConfig: {
    `vector.dimensions`: 1536,
    `vector.similarity_function`: 'cosine'
  }
}

Querying by Vector Similarity

// Find the 10 most similar chunks to a query embedding
CALL db.index.vector.queryNodes(
  'chunk_embeddings',    // index name
  10,                       // top-k results
  $queryEmbedding           // the query vector
)
YIELD node AS chunk, score
RETURN chunk.text, score
ORDER BY score DESC
Supported configurations Vector dimensions up to 4096. Similarity functions: cosine, euclidean, and dot (since Neo4j 5.18). Compatible with embeddings from OpenAI, Anthropic (Voyage), Cohere, and open-source models.

4. Hybrid Retrieval: The Real Power

The defining advantage is combining vector similarity with graph traversal in a single query. This is variously called "GraphRAG" or "hybrid retrieval."

Pattern: Vector Search + Graph Context Expansion

// Step 1: Find semantically similar chunks
CALL db.index.vector.queryNodes('chunk_embeddings', 5, $queryEmbedding)
YIELD node AS chunk, score

// Step 2: Traverse the graph to enrich with structured context
MATCH (chunk)-[:MENTIONS]->(entity)
OPTIONAL MATCH (entity)-[:REGULATED_BY]->(reg:Regulation)
OPTIONAL MATCH (chunk)-[:CHUNK_OF]->(doc:Document)

RETURN
  chunk.text       AS chunkText,
  score            AS similarity,
  doc.title        AS sourceDocument,
  collect(entity)  AS relatedEntities,
  collect(reg)     AS applicableRegulations

This single query does what would otherwise require two separate systems: it finds semantically relevant text and expands into the structured knowledge graph to pull in related entities, regulations, and provenance.

Pattern: Conversation-Aware Retrieval

// Find past conversation context relevant to the current query
CALL db.index.vector.queryNodes('message_embeddings', 5, $queryEmbedding)
YIELD node AS msg, score
WHERE score > 0.75

// Get the conversation thread and any entities discussed
MATCH (msg)-[:PART_OF]->(conv:Conversation)
MATCH (msg)-[:REFERENCES]->(entity)
OPTIONAL MATCH (msg)-[:FOLLOWS]->(prev:Message)

RETURN
  msg.content      AS messageText,
  conv.title       AS conversationTitle,
  prev.content     AS precedingMessage,
  collect(entity)  AS discussedEntities,
  score            AS relevance
Why this matters for agents An agent answering a question can retrieve semantically similar past conversations and follow the graph links to see what domain entities were discussed, what decisions were made, and what the structured state of those entities is now. The vector search finds relevance; the graph provides truth.

5. Comparison: Single-Store vs. Dual-Store Architecture

Concern Neo4j Hybrid Separate Graph + Vector DB
Data consistency Single transaction boundary Eventual consistency; sync lag risk
Query capability Vector + graph in one Cypher query Two queries, application-level join
Operational complexity One database to deploy, back up, monitor Two systems with independent failure modes
Vector search performance Good (HNSW); dedicated vector DBs may edge ahead at very large scale Optimised for vector-only workloads
Graph traversal performance Native, index-free adjacency Not available in vector DB
Context enrichment Immediate — traverse from vector result node Requires a second lookup by ID in the graph DB
Access control Graph-native permissions apply to both Must enforce in two places
When to prefer the hybrid approach When your retrieval needs to combine semantic similarity with structural relationships — e.g., "find relevant safety documents for this specific project's jurisdiction." The graph narrows the search space; the vector finds relevance within it.

6. Practical Architecture

User / Agent | v Application Layer | |—— Embed query text (OpenAI / Voyage / local model) |—— Send embedding + filters to Neo4j | v +-------------------------------------------------+ | Neo4j | | | | Vector Index (HNSW) | | • Chunk embeddings | | • Message embeddings | | • Document embeddings | | | | Knowledge Graph | | • Domain entities + relationships | | • Document → Chunk → Entity links | | • Conversation → Message → Entity links | | • Temporal metadata (valid_from / valid_until)| | • Provenance (source citations, actor trails) | | | | Hybrid Query Engine | | • Single Cypher query combines both | | • Vector similarity + graph traversal | | • Filtered by graph-native permissions | +-------------------------------------------------+

Ingestion Pipeline

  1. Document arrives — PDF, HTML, or structured data
  2. Chunk — split into semantically meaningful segments
  3. Embed — generate vector embedding for each chunk
  4. Extract entities — identify domain entities mentioned (NER or LLM extraction)
  5. Store in Neo4j — create Chunk nodes with embedding property, link to Document node, link to extracted Entity nodes via MENTIONS relationships
  6. Index — the vector index automatically picks up new embedding properties

Conversation Storage Pipeline

  1. Message received — from user or agent
  2. Embed — generate vector embedding of the message content
  3. Extract references — identify domain entities mentioned
  4. Store in Neo4j — create Message node with embedding, link to Conversation, User/Agent, referenced Entities, and preceding Message

7. Advanced Patterns

Scoped Vector Search

Pre-filter the vector search using graph relationships, so only contextually relevant nodes are searched. This dramatically improves precision. 

// Only search chunks from documents relevant to a specific project
MATCH (p:Project {id: $projectId})-[:GOVERNED_BY]->(reg:Regulation)
MATCH (chunk:Chunk)-[:MENTIONS]->(reg)
WITH collect(chunk) AS candidates

// Vector search within the pre-filtered set
CALL db.index.vector.queryNodes('chunk_embeddings', 10, $queryEmbedding)
YIELD node AS chunk, score
WHERE chunk IN candidates

RETURN chunk.text, score

Multi-Hop Context Expansion

After finding relevant chunks via vector search, traverse multiple hops through the graph to build a rich context window for the LLM. 

// Find similar chunks, then expand 2 hops into the knowledge graph
CALL db.index.vector.queryNodes('chunk_embeddings', 3, $queryEmbedding)
YIELD node AS chunk, score

// Expand: chunk → entity → related entities (2 hops)
MATCH path = (chunk)-[:MENTIONS]->(e1)-[*1..2]-(e2)
WHERE NONE(n IN nodes(path) WHERE n:Chunk AND n <> chunk)

RETURN
  chunk.text AS sourceText,
  score,
  [n IN nodes(path) | labels(n)[0] + ': ' + n.name] AS contextPath

Temporal Filtering

Combine vector similarity with temporal validity to ensure only currently-active knowledge is retrieved. 

CALL db.index.vector.queryNodes('chunk_embeddings', 10, $queryEmbedding)
YIELD node AS chunk, score

MATCH (chunk)-[:MENTIONS]->(reg:Regulation)
WHERE reg.valid_from <= date() AND reg.valid_until >= date()

RETURN chunk.text, reg.name, reg.source, score

9. When to Use This Pattern

Good Fit

  • RAG where context requires structured relationships
  • Conversational agents with domain-specific knowledge
  • Compliance / regulatory systems with document references
  • Multi-tenant applications needing unified access control
  • Systems where provenance and auditability matter

Consider Alternatives

  • Pure vector search at extreme scale (billions of vectors)
  • Simple semantic search without relationship context
  • Workloads that are 99% vector, 1% graph
  • Cost-constrained deployments where a lightweight vector DB suffices
